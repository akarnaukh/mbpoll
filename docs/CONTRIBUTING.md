## Как вы можете помочь

Есть множество способов внести вклад в проект:

- **Сообщение об ошибках** — найдя баг, создайте issue
- **Предложение функций** — предложите новую функцию через issue
- **Исправление ошибок** — форкните репозиторий и отправьте PR

- **Тестирование** — протестируйте новые функции и сообщите о результатах

## Внесение кода

### Шаг 1: Форк репозитория

```bash
git clone https://github.com/your-username/mbpoll.git
cd mbpoll
git remote add upstream https://github.com/akarnaukh/mbpoll.git
```

### Шаг 2: Создание ветки

```bash
# Обновите main ветку
git fetch upstream
git checkout main
git merge upstream/main

# Создайте ветку для вашей функции/исправления
git checkout -b feature/your-feature-name
# или
git checkout -b fix/issue-123
```

### Шаг 3: Внесение изменений

```bash
# Редактируйте файлы
nano src/your_file.c

# Проверка стиля кода
make check-style

# Сборка
make all

# Тестирование
make test
```

### Шаг 4: Коммиты

```bash
git add src/your_file.c
git commit -m "fix: описание исправления

Детальное описание изменений
- Что было исправлено
- Почему это важно
- Ссылка на issue (если есть)

Fixes #123"
```

### Формат сообщений коммитов

Мы используем [Conventional Commits](https://www.conventionalcommits.org/):

- `feat:` — новая функция
- `fix:` — исправление ошибки
- `docs:` — изменения в документации
- `style:` — форматирование, отступы
- `refactor:` — рефакторинг кода
- `test:` — добавление тестов
- `chore:` — служебные изменения

**Примеры:**
```
feat: добавить поддержку Modbus FC 23
fix: исправить утечку памяти при переподключении
docs: обновить README с примерами конфигурации
refactor: упростить обработку ошибок WebSocket
```

### Шаг 5: Отправка PR

```bash
git push origin feature/your-feature-name
```

1. Перейдите на GitHub
2. Нажмите "Compare & pull request"
3. Заполните описание PR
4. Отправьте на ревью

## Стандарты кода

### C Code Style

Мы следуем стандартам кодирования на языке C:

#### Именование

```c
// Функции: snake_case
int load_config(const char *filename, config_t *config);
void broadcast_to_websockets(config_t *config, const char *data);

// Переменные: snake_case
int poll_interval_ms;
char device_address[256];

// Структуры: snake_case_t
typedef struct {
    int socket_fd;
    char ip_address[32];
} websocket_client_t;

// Константы: UPPER_CASE
#define MAX_DEVICES 256
#define DEFAULT_PORT 502

// Макросы: UPPER_CASE
#define LOG_ERROR(msg) log_error(__FILE__, __LINE__, msg)
```

#### Форматирование

```c
// Отступы: 4 пробела
if (condition) {
    do_something();
    do_another_thing();
}

// Пробелы вокруг операторов
int result = value1 + value2;
if (x > 0 && x < MAX_VALUE) {
    // ...
}

// Пустые строки между функциями
void function_one() {
    // ...
}

void function_two() {
    // ...
}
```

#### Комментарии

```c
// Однострочные комментарии с пробелом после //

/*
 * Многострочные комментарии
 * для описания сложных блоков
 */

/**
 * @brief Краткое описание функции
 * @param param1 Описание параметра
 * @return Описание возвращаемого значения
 */
int process_data(int param1);
```

#### Обработка ошибок

```c
// Всегда проверяйте возвращаемые значения
int result = some_function();
if (result != 0) {
    log_error("Function failed with code %d", result);
    return result;
}

// Освобождайте ресурсы при ошибке
if (allocation_failed) {
    free(allocated_memory);
    close(opened_fd);
    return ERROR_CODE;
}
```

#### Память

```c
// Выделение памяти
config_t *config = malloc(sizeof(config_t));
if (!config) {
    log_error("Failed to allocate memory");
    return EXIT_FAILURE;
}
memset(config, 0, sizeof(config_t));

// Освобождение памяти
free(config);
config = NULL;  // Избегаем висячих указателей
```

### Проверка стиля

```bash
# Установка cpplint
pip install cpplint

# Проверка стиля
cpplint --extensions=c,h src/*.c src/*.h

# Или используйте make
make check-style
```

## Тестирование

### Типы тестов

1. **Модульные тесты** — тестирование отдельных функций
2. **Интеграционные тесты** — тестирование взаимодействия компонентов
3. **Системные тесты** —端到端 тестирование

### Запуск тестов

```bash
# Все тесты
make test

# Модульные тесты
make test-unit

# Интеграционные тесты
make test-integration

# Тесты производительности
make test-bench
```

### Написание тестов

```c
#include <assert.h>
#include "config.h"

void test_load_config_valid() {
    config_t *config = malloc(sizeof(config_t));
    
    int result = load_config("test.conf", config);
    
    assert(result == 0);
    assert(config->poll_interval_ms == 5000);
    assert(config->websocket_port == 24123);
    
    free(config);
}

void test_load_config_invalid() {
    config_t *config = malloc(sizeof(config_t));
    
    int result = load_config("nonexistent.conf", config);
    
    assert(result != 0);
    
    free(config);
}

int main() {
    test_load_config_valid();
    test_load_config_invalid();
    printf("All tests passed!\n");
    return 0;
}
```

### Тестирование с реальными устройствами

Если у вас есть доступ к Modbus устройствам:

```bash
# Настройка тестового окружения
./setup_test_environment.sh

# Запуск тестов с устройствами
./test_with_devices.sh
```

## Процесс ревью

### Чеклист перед отправкой PR

- [ ] Код следует стандартам проекта
- [ ] Все тесты проходят
- [ ] Документация обновлена
- [ ] Коммиты имеют понятные сообщения
- [ ] Нет лишних изменений (whitespace, и т.д.)
- [ ] Добавлены тесты для новых функций

### Что происходит после отправки PR

1. **Автоматические проверки** — CI запускает тесты
2. **Ревью от мейнтейнеров** — проверка кода и архитектуры
3. **Обсуждение** — комментарии и предложения по улучшению
4. **Изменения** — внесение правок по замечаниям
5. **Мерж** — принятие в основную ветку

### Время ответа

Мы стараемся отвечать на PR в течение:
- Исправления ошибок: 1-3 дня
- Новые функции: 3-7 дней
- Документация: 1-5 дней

### Причины отклонения PR

- Не соответствует архитектуре проекта
- Нет тестов для новых функций
- Нарушает обратную совместимость без необходимости
- Недостаточно документировано
- Дублирует существующую функциональность
